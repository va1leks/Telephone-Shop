## Введение

Данный документ описывает план тестирования приложения "Telephone Shop" (Фронтенд на React, Бэкенд на Spring Boot). Документ предназначен для людей, выполняющих тестирование данного проекта. 
Цель тестирования — проверка соответствия реального поведения приложения и его ожидаемого поведения, описанного в бизнес-требованиях и предыдущих технических рекомендациях.

<a name="items"/>

## Объект тестирования

В качестве объектов тестирования можно выделить следующие функциональные требования, реализуемые через REST API и пользовательский интерфейс:

* **Просмотр каталога товаров:** Отображение списка всех телефонов.
* **Карточка товара:** Просмотр детальной информации о конкретном телефоне.
* **Поиск и фильтрация:** Поиск по текстовому запросу и фильтрация по характеристикам (бренд, цена, модель).
* **Управление корзиной:** Добавление товара в корзину конкретного клиента.
* **Создание заказа:** Оформление заказа на основе содержимого корзины клиента.

<a name="quality"/>

## Атрибуты качества

1.  **Функциональность:**
    * Функциональная полнота: Приложение должно выполнять все заявленные функции, включая работу с изображениями.
    * Функциональная корректность: Операции с корзиной и заказами должны корректно сохраняться в базе данных (целостность данных).
2.  **Производительность:**
    * Скорость отклика: API-эндпоинты (особенно `/filter`) должны возвращать результат за минимальное время (не более 1-2 секунд) при стандартной нагрузке.
3.  **Удобство использования (Usability):**
    * Интуитивность: Процесс добавления товара в корзину и создания заказа должен быть интуитивно понятным.

<a name="risk"/>

## Риски

К рискам, связанным с данным проектом, можно отнести:
* **Ошибка CORS:** Блокировка запросов с фронтенда на бэкенд из-за некорректной настройки Cross-Origin Resource Sharing.
* **Нарушение целостности данных:** Попытка создать заказ или добавить в корзину товар, ID которого отсутствует в главной таблице `telephone` (как уже наблюдалось в логах).
* **Проблема с файлами (изображениями):** Изображения могут не загружаться или отображаться некорректно в браузере.

<a name="features"/>

## Аспекты тестирования

В ходе тестирования планируется проверить реализацию основных функций приложения.

### 1. Просмотр и фильтрация каталога
* Запуск фронтенда и бэкенда (проверка связи).
* Отображение полного списка телефонов на главной странице.
* Переход к карточке товара по клику.
* Проверка работы фильтрации по бренду (например, "Apple").
* Проверка работы текстового поиска по части названия модели.
* Проверка совместной работы фильтрации и поиска.

### 2. Карточка товара и корзина
* Отображение всех характеристик (модель, цена, описание) на детальной странице.
* Успешное выполнение запроса на добавление товара в корзину клиента.
* Проверка, что в корзину нельзя добавить несуществующий товар (Негативный тест).
* Проверка, что корзина отображает только товары для нужного клиента.

### 3. Создание заказа
* Успешное выполнение запроса на создание заказа.
* Проверка, что после создания заказа статус заказа установлен как "Pending".
* Проверка, что товар(ы) из корзины клиента корректно переместились в сущности заказа (`AvailableTelephone`).
* Проверка, что корзина очищается после успешного создания заказа (через отдельный API вызов или автоматически).

### 4. API изображений
* Проверка загрузки превью-изображения на главной странице.
* Проверка загрузки всех изображений на детальной странице.
* Проверка, что эндпоинт `/images/{id}` возвращает корректный `Content-Type` (image/jpeg, image/png).

<a name="approach"/>

## Подходы к тестированию

При тестировании будет использован ручной подход:
* **Тестирование API:** Используя инструменты (например, Postman) для проверки эндпоинтов бэкенда.
* **Тестирование UI:** Проверка взаимодействия через веб-интерфейс (React).
